Main data views

View tasks

This screen shows you a set of filters, and matching tasks. You can filter by:

  • any one of the ID numbers;
  • surname, forename, DOB, sex;
  • task type;
  • whether the task was properly completed;
  • the device and/or user that uploaded the task;
  • a start and/or end date, to constrain the tasks to a date range;
  • any text found in the fields for the task.
    • Hint: e.g. type in “paracetamol” to find clerkings that mention paracetamol anywhere.

You may view each task in two formats:

  • HTML: this will be served to you fastest, so is useful for quick viewing, but is not suitable for printing in a clinical environment (because it won’t have patient identifiers on each page).
    • Hint: At the bottom of the HTML, there will be a link to view the PDF (and perhaps links to view older/newer versions of the same task, if it has been edited and re-uploaded; the newest will always be shown first).
    • Superusers have some additional options; see Administration.
  • PDF: this is suitable for printing, with identifiers on each page and a signature block for clinician-administered tasks (but it takes slightly longer to appear, as the server has to generate it first).
    • Here are some fictional examples: PHQ-9; psychiatric clerking (albeit not a very good one!).
    • Security hint: some tablet browsers (e.g. Chrome for Android) save PDFs to the tablet by default. In that case, avoid viewing PDFs from that browser, or you’ll have to clean the patient-identifiable PDF off the tablet yourself.

Trackers

Trackers show quantities over time. They work like this:

  • you choose a patient (using one of the ID numbers);
  • optionally, you specify a start and/or end date;
  • you choose one or more or all tasks (from those that support trackers);
  • you specify an HTML (quick) or a PDF (print-formatted) view.

Then:

  • All matching tasks are found. The tracker is generated.
  • A consistency check is performed and added to the tracker to ensure that names, dates of birth, sexes, and all other ID numbers are consistent in all the tasks. Beware if the consistency check fails.

Clinical text views

Whereas trackers show numerical information, clinical text views show clinically relevant textual information over time.

  • you choose a patient (using one of the ID numbers);
  • optionally, you specify a start and/or end date;
  • you specify an HTML (quick) or a PDF (print-formatted) view.

Then:

  • All matching tasks are found. The view is generated.
  • A consistency check is performed and added to the tracker to ensure that names, dates of birth, sexes, and all other ID numbers are consistent in all the tasks. Beware if the consistency check fails.
  • Tasks providing primary clinical text (e.g. diagnostic coding, psychiatric clerkings, and progress notes) show that information directly.
  • Other tasks are noted in the chronology (and in the HTML version of the view, are hyperlinked to).

Reports and data dumps (for authorized users)

These options are not visible to most users.

Run reports

This provides a selection of built-in reports that you can run. Output is offered in HTML (for easy viewing) or tab-separated values (TSV) format for downloading and importing into spreadsheets and the like.

Basic research dump (fields and summaries)

This should be your starting point for a research dump. It’s not quite as powerful as the table/view data dump (see below), but it’s fairly comprehensive and much simpler to use. This option provides tab-separated value (TSV) files for each task (occasionally more than one per task, but usually one) in a ZIP file. By default it is restricted to current tasks (generally helpful as you don’t often want to view the edit history of all tasks). It incorporates patient and summary information for you, and the summaries are calculated “live” (unlike the table/view data dump, below, which requires you to regenerate summary tables first). So that’s also helpful. The configuration screen provides plenty of explanation, along with examples of how to import the resulting files into different software packages. You can choose to download:

  • Everything (all current records);
  • those tasks specified by your current task filters (e.g. to restrict by patient, start/end date, and/or a single task type, etc.);
  • just specific tasks (all current records for tasks that you choose).

Regenerate summary tables

This wipes and regenerates the summary tables. Use it before dumping summary data with the table/view data dump option (below).

Dump table/view data

This allows authorized (highly privileged) users to download task data (with summary information), BLOBs (binary large objects, such as pictures), and patient information. The data is in the same format as the underlying database; that is, more powerful but more complex than the basic research dump. The SQL option allows you to recreate a copy of the database. With this option, views are downloaded as view definitions, just as in the original database. The TSV option downloads views as their content, which is often convenient. (Beware downloading the BLOBs. If you insist on downloading the BLOBs, they will be hex-encoded in the TSV format, with the MySQL HEX() function.)

Inspect table definitions (with or without view definitions)

These options allow you to view the table structure in use on the server. Most significantly, you can see the field comments, which give details of each field. Summary tables will only be included if you’ve generated them (see above).

Administrative options (administrators only)

Manage users

This allows administrators to add and delete users, reset their passwords, and edit their permissions.

The permissions are:

  • May use web viewer: allows access to this web viewer. Usually YES.
  • May view other users’ records: allows the user to see tasks uploaded by others. Usually YES (beware saying NO: there is a significant clinical risk to being unable to see data!).
  • Sees all patients’ records when unfiltered: usually NO, for reasons of confidentiality; this is usually a superuser permission (though it may slow the webview down when no patient is selected).
  • May upload data from tablet devices: usually YES, or the user will be unable to upload data from tablets; however, NO may be correct for administrative/support staff.
  • May manage users: usually NO; this defines superuser authority.
  • May register tablet devices: depends on your local security policy (if you say NO, an administrator will have to register all users’ tablets).
  • May use mobileweb storage facility: a facility in development, so say NO for now.
  • May dump data: ability to download raw data (typically YES for authorized researchers and NO for all others). Users with this permission can see raw data from the entire database regardless of other settings.
  • May run reports: ability to run prespecified reports (typically YES for selected clinicians and administrators). Users with this permission can run reports based on data from the entire database regardless of other settings.

Delete patient entirely

This completely destroys all data associated with a patient, as defined by a specific ID number. Use it e.g. when the records are old enough that they must be destroyed. Use it with great care!

Note: you can erase data for individual tasks, or apply a special note to all versions of a task, using the superuser options on that task’s HTML view. See Administration.

View audit trail

Explore the recent (or not so recent) audit log.

View HL7 message log

Shows recent tasks exported to different recipients via the HL7 or file export system. Each message is associated with a run (see below).

View HL7 run log

Shows recent HL7/file export runs. Note that many runs may send no messages, because there were no new tasks to export.

Other actions

Show server identification policies

This displays:

  • the server’s long and short descriptions of all the ID numbers — see Patient Identification
  • the server’s “upload” and “finalize” ID policies — see Patient Identification

Introspect source code (if enabled)

If enabled on the server, this allows you to browse the current source code that the server is running, and the most recent tablet source code available when the server software was built. You may find this option helpful for understanding the internal workings of each task, e.g. scoring systems, or simply of interest. If your administrator has not enabled introspection, the source code is available via www.camcops.org.
